# daily-market-radar｜每日執行最小閘門

本檔用來測試「少數關鍵硬閘門」是否能改善每日播報執行品質。

原則：不要新增複雜流程，只把既有 Loop / 強制規則濃縮成三個不可省略的輸出欄位。

---

## 1. 三個最小閘門

每日播報輸出前，只強制確認三件事：

```text
1. 硬閘門狀態
2. 6 大核心領域 Coverage Matrix
3. 高風險 claim 檢查表
```

若這三件事沒有輸出，不得稱為完整正式播報。

---

## 2. 硬閘門狀態

每日報告最上方必須明確輸出：

```text
每日訊號硬閘門狀態：通過 / 未通過 / 搜尋未完整 / 歷史去重未完整
```

判定：

- 6 大核心領域皆達 5 則大型重要新聞 + 3 則小眾潛力候選，且已完成歷史去重與必要 retry，才能標示「通過」。
- 任一領域未達 5 + 3，標示「未通過」。
- 未完整搜尋或未完整讀取必要 repo 檔案，標示「搜尋未完整」。
- 未完整檢查最近 7 日 reports，標示「歷史去重未完整」。

若狀態不是「通過」，不得稱為完整正式播報。

---

## 3. Coverage Matrix

每日報告前段必須輸出：

| 核心領域 | 大型新聞數 | 小眾候選數 | 是否達標 | 缺口 |
|---|---:|---:|---|---|
| AI 模型 / Agent / 工作流替代 |  |  |  |  |
| 區塊鏈 / 加密 / RWA / Agent payments |  |  |  |  |
| 零售 / 消費 / 社群 / 服飾 |  |  |  |  |
| 全球市場 / 資金流 / 地緣政治 |  |  |  |  |
| 科技發展 / 機器人 / 生技 / 能源 / 半導體 |  |  |  |  |
| 勞動 / 消費壓力 / 台灣本地訊號 |  |  |  |  |

判定：

```text
大型新聞數 >= 5 且小眾候選數 >= 3 → 達標
大型新聞數 < 5 或小眾候選數 < 3 → 未達標
```

不得用「濃縮版達標」「大型新聞已涵蓋小眾候選」「小眾候選已涵蓋大型新聞」替代數量檢查。

---

## 4. 高風險 claim 檢查表

不需要每一則新聞都拆 claim table，只檢查高風險 claim。

高風險 claim 包含：

- 精確數字。
- 重大政策。
- 公司重大事件。
- 新技術突破。
- 加密資金流。
- 台灣本地判斷。
- 外部模型提供的陌生訊號。

每日報告若包含高風險 claim，需輸出：

| Claim | 來源 / 可回查線索 | 來源時間 | 證據等級 | 採用狀態 | 處理方式 |
|---|---|---|---|---|---|
|  |  |  | 高 / 中 / 低 / 資料不足 | 採用 / 降級 / 不採用 |  |

判定：

- 有官方、權威媒體或多來源交叉驗證：可採用。
- 有可信媒體但缺完整數據：可採用，但標中證據與不確定點。
- 只有社群或模型敘事：只能列候選。
- 查不到來源：不得放入事實區。
- 來源支持方向但不支持數字：保留方向，數字降級或不採用。
- 來源支持舊事件但不是今日新事件：標示為背景訊號或歷史已播。

---

## 5. 失敗輸出格式

若任一閘門未通過，報告必須寫：

```text
本次報告未通過每日執行最小閘門。
不可視為完整正式播報。
未通過原因：
1. ...
2. ...
3. ...
下一步補查：
1. ...
2. ...
3. ...
```

不得用漂亮總結掩蓋未完成狀態。

---

## 6. 外部模型輸出規則

外部模型輸出不得直接併入正式報告。

必須先經過：

```text
外部模型輸出 → 拆成高風險 claim → 查來源 → 證據分級 → 採用 / 降級 / 不採用 → 回補到報告或規則
```

外部模型適合當 scout，不可直接當 judge。
